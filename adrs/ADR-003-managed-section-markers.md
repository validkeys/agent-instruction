# ADR-003: Managed Section Markers

**Status**: Accepted

**Date**: 2026-04-13

**Deciders**: Development Team

---

## Context

The agent-instruction tool needs to update instruction files (CLAUDE.md, AGENTS.md) while preserving user-written content. Users should be able to:

1. **Add custom content** to instruction files (project-specific guidelines, notes, etc.)
2. **Preserve their content** when the tool regenerates instructions
3. **Clearly see** which sections are tool-managed vs user-controlled
4. **Edit user sections** without tool interference

The challenge: How do we delineate tool-managed sections from user content in a way that is:
- Visible and understandable to humans
- Reliably parseable by the tool
- Non-invasive to the document structure
- Works with Markdown syntax

## Decision

We will use **HTML comment markers** to delineate managed sections:

```markdown
# User's Custom Instructions

This is user-written content that the tool never modifies.

<!-- BEGIN AGENT-INSTRUCTION -->
# Generated Instructions

This content is generated by `agent-instruction build` and will
be replaced on every build. Do not manually edit.

## Rule: Error Handling
- Always use fmt.Errorf with %w for error wrapping
<!-- END AGENT-INSTRUCTION -->

More user content here is preserved.
```

**Marker Format:**
```go
const (
    BeginMarker = "<!-- BEGIN AGENT-INSTRUCTION -->"
    EndMarker   = "<!-- END AGENT-INSTRUCTION -->"
)
```

**Behavior:**
- Content before `BEGIN` marker: Preserved (user-controlled)
- Content between markers: Replaced on every build (tool-managed)
- Content after `END` marker: Preserved (user-controlled)
- If no markers exist: Entire file is user-controlled, markers added on first build

## Alternatives Considered

### Alternative 1: Front Matter with Delimiter
```yaml
---
managed: true
generator: agent-instruction
---
# Generated content below
```

**Pros:**
- Common pattern (used by Jekyll, Hugo)
- Structured metadata
- Easy to parse

**Cons:**
- Only works for top of file
- Can't preserve user content before generated content
- Requires YAML parser
- Not visible when viewing file (at top, often collapsed)

**Rejected because:** Doesn't support preserving user content before and after managed section.

### Alternative 2: Separator Comments
```markdown
<!-- agent-instruction:start -->
Generated content
<!-- agent-instruction:end -->
```

**Pros:**
- Shorter syntax
- Familiar pattern (similar to code generation tools)

**Cons:**
- Less explicit about what happens in the section
- "start/end" less clear than "begin/end"
- Doesn't explicitly warn users not to edit

**Rejected because:** Less clear intent. "BEGIN AGENT-INSTRUCTION" is more explicit than "start".

### Alternative 3: Separate File
```
CLAUDE.md           <- User-written
CLAUDE.generated.md <- Tool-generated
```

Tool would concatenate or include generated file.

**Pros:**
- Complete separation
- No parsing needed
- Very clear ownership

**Cons:**
- Two files to manage
- Concatenation order unclear (before or after user content?)
- Users must remember to reference both files
- Breaks single-file convention

**Rejected because:** Creates complexity and breaks the single-file paradigm.

### Alternative 4: Indented Block Quotes
```markdown
> BEGIN AGENT-INSTRUCTION
> # Generated Content
> This is tool-managed
> END AGENT-INSTRUCTION
```

**Pros:**
- Visible in rendered Markdown (quoted)
- No HTML comments
- Pure Markdown

**Cons:**
- Renders as quoted block (looks wrong)
- Harder to parse (need to handle indentation)
- Markdown renderers vary in quote handling
- Looks unprofessional

**Rejected because:** Visual appearance is poor, and parsing is complex.

### Alternative 5: Fence Blocks
````markdown
```agent-instruction
# Generated content
```
````

**Pros:**
- Markdown-native fence syntax
- Good syntax highlighting

**Cons:**
- Renders as code block (completely wrong)
- Content appears as code, not instructions
- Not suitable for prose content
- Confusing for readers

**Rejected because:** Completely wrong semantic meaning.

## Consequences

### Positive

1. **User Content Preserved**
   - Users can add content before managed section (preamble, project context)
   - Users can add content after managed section (additional notes)
   - Tool never touches user sections

2. **Clear Ownership**
   - HTML comments are visible in plain text
   - Marker names explicitly state "AGENT-INSTRUCTION"
   - BEGIN/END clearly delineate boundaries
   - Intent is obvious: "This section is managed"

3. **Markdown Compatible**
   - HTML comments are valid in Markdown
   - Don't affect rendered output
   - Work with all Markdown renderers (GitHub, GitLab, etc.)
   - Invisible when viewed in rendered form

4. **Reliable Parsing**
   - Simple string search: `strings.Index(content, BeginMarker)`
   - No regex complexity
   - No YAML parsing needed
   - Deterministic behavior

5. **Multiple Sections Possible**
   - Could support multiple managed sections per file (future)
   - Current implementation validates exactly one section

### Negative

1. **Markers Visible in Plain Text**
   - HTML comments appear when viewing raw Markdown
   - Mitigation: Clear, understandable names

2. **Manual Editing Risk**
   - Users could edit content between markers
   - Changes would be lost on next build
   - Mitigation: Marker names warn users ("AGENT-INSTRUCTION")

3. **Marker Corruption**
   - Users could delete one marker
   - Tool detects and returns error (won't corrupt file)
   - Mitigation: Validation ensures both markers present

### Neutral

1. **HTML Comment Choice**
   - Could have used other formats (YAML, custom tags)
   - HTML comments chosen for Markdown compatibility
   - Familiar to developers

2. **Marker Naming**
   - "AGENT-INSTRUCTION" is specific to this tool
   - Could have used generic name
   - Specific name makes intent clearer

## Implementation Notes

### Parsing Logic (managed.go)
```go
func ParseManagedContent(content string) (*ManagedContent, error) {
    beginIdx := strings.Index(content, BeginMarker)
    endIdx := strings.Index(content, EndMarker)

    // No markers - all user content
    if beginIdx == -1 && endIdx == -1 {
        return &ManagedContent{Before: content}, nil
    }

    // Validation: both markers must be present
    if beginIdx == -1 || endIdx == -1 {
        return nil, fmt.Errorf("malformed managed section")
    }

    // Validation: correct order
    if beginIdx > endIdx {
        return nil, fmt.Errorf("end marker appears before begin marker")
    }

    // Extract sections
    before := content[:beginIdx]
    managed := content[beginIdx+len(BeginMarker):endIdx]
    after := content[endIdx+len(EndMarker):]

    return &ManagedContent{Before: before, Managed: managed, After: after}, nil
}
```

### Replacement Logic (managed.go)
```go
func ReplaceManagedSection(content string, newSection string) (string, error) {
    // If no markers, append with markers
    if !HasManagedSection(content) {
        return content + "\n\n" + BeginMarker + "\n" + newSection + "\n" + EndMarker, nil
    }

    // Replace content between markers
    before := content[:beginIdx+len(BeginMarker)]
    after := content[endIdx:]
    return before + "\n" + newSection + "\n" + after, nil
}
```

### Error Handling
Tool validates markers before modifying files:
- Missing begin marker: Error, won't modify file
- Missing end marker: Error, won't modify file
- Wrong order: Error, won't modify file
- User can fix markers manually before retrying

### Backup Strategy
Before modifying file with managed section, tool creates `.backup` file (see backup.go). If something goes wrong, user has recovery option.

## Testing Strategy

Comprehensive tests in `internal/files/managed_test.go`:
- Parse content with managed section
- Parse content without markers
- Parse malformed markers (one missing, wrong order)
- Replace existing managed section
- Add managed section to unmarked file
- Preserve whitespace and formatting
- Multiple lines in managed section

## User Documentation

Documentation should clearly explain:
1. What managed sections are
2. How to identify them (HTML comments)
3. That content between markers will be replaced
4. How to add custom content (before or after markers)
5. What happens if markers are corrupted

## Future Enhancements

Potential future features (not implemented now):
1. **Named sections**: `<!-- BEGIN AGENT-INSTRUCTION:rules -->`
2. **Multiple sections**: Support more than one managed section per file
3. **Marker customization**: Allow users to customize marker format (low priority)
